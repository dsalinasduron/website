An abstract data type - ADT - is

a specification that describes a data set and a set of operations on that data.

An ADT is conceptual and describes the behavior of an object. It does not indicate how to store the data or how the operations are implemented.

Objects that store and manipulate data are called data structures. An ADT is a technique for describing the conceptual behavior of a data structure, without defining how it is implemented.

Example - Bag

A Bag is an ADT that holds a collection of items. Some of its behavior:

- there can be duplicate items

- in real-life, items are heterogeneous

- in this ADT, items must be homogeneous (they must all be of the same type)

- items have no order

- it may or may not have a finite capacity (our initial implementation will have a finite capacity)

There are in fact alternatives to arrays for implementing the Bag interface. Assume we were to use one of these alternatives: apart from changing the name of the object instance from ArrayBag to another appropriate name, would the above unit tests still be appropriate?

Polymorphism allows us to easily change instance representations. In a short while we will look at other implementations of the Bag interface. Polymorphism allows us to replace

        Bag bag = new ArrayBag(5);
       
   bag.add("Apple");
   bag.add("Banana");
   bag.add("Cherry");
   bag.add("Dates");
   bag.add("Eggplant");

with

        Bag bag = new ArrayListBag(5);
       
   bag.add("Apple");
   bag.add("Banana");
   bag.add("Cherry");
   bag.add("Dates");
   bag.add("Eggplant");

The behavior of the bag remains the same, its implementation has changed. This is the fundamental idea behind an abstract data type and how polymorphism allows us to readily substitute one implementation for another.
