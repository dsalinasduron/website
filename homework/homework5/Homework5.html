    At a fundamental level this is what is occurring:

    Asteroids a = new Asteroids();
    a.repaint();
    a.paint(...)
    a.paint(...)
    a.paint(...)
    a.paint(...)
    .

This paragraph will explain the details of what is actually happening when the main() method calls a.repaint(), which is a method defined by Java in the class Component, which is the superclass of Canvas (which is the superclass of Game). The method a.repaint() calls the method a.update() over and over again. The method update() is defined in the superclass Game (overriding the original definition in Component), and it first calls the method a.paint(...), which you will fill in in Asteroids. An instance of Graphics is passed into a.paint(...), but this is not the instance that holds the game Canvas. Instead, this instance of Graphics acts as a buffer (and hence is called buffer). So when your code in a.paint(...) draws something, it is actually drawing it on this buffer Graphics instance. Everything in the buffer is then drawn to the screen all at once, making the animation less choppy. A schematic of what is really happening is below:

Asteroids a = new Asteroids();
a.paint(...) -> draws to buffer
draw buffer on screen
a.paint(...) -> draws to buffer
draw buffer on screen
a.paint(...) -> draws to buffer
draw buffer on screen

Detailed Instructions:

The spaceship and asteroids are all going to be drawn on screen as polygons. Recall that a polygon is a chain of line segments that close to form a loop. Some examples of polygons are shown below:

For this first milestone, you will create an Asteroid class that extends Polygon. (An Asteroid is-a Polygon.) You will then create an Asteroid object in Asteroids and display it on the screen.

Look at the class Polygon. An instance of this class represents a polygon, and its instance variables store the following information:

• the shape of the Polygon, represented as an array of Point instances corresponding to the vertices (corners)

• the offset between the origin and the center of the Polygon instance, represented as a Point 

• the rotation in degrees of Polygon instance from its starting orientation (as given by the shape array)

Take notice of the instance variable names in Polygon .

You’ll notice that coordinates are represented by instances of the Point class. Look at the Point class to see what its instance variables are, and what methods it has available.

Your brush object is a Java Graphics object, where the Graphics class has a drawPolygon() method. The move() method is used to move the Asteroid across the canvas. This can be done by adjusting the offset in the Polygon class (represented by instance data position).

We will use a method in the class Graphics (https://docs.oracle.com/javase/7/docs/api/java/awt/Graphics.html

) to draw the Polygon instance. All of the methods in Graphics are non-static, so we call them by brush.nameOfMethod(..), where brush is the parameter of our paint() method in Asteroid.

Drawing in Java graphics uses the Cartesian coordinate system, but it is different than what you might expect. Imagine a graphics layout that is 800 pixels wide by 600 pixels tall. The (0,0) origin is in the upper left hand corner:

To draw an Asteroid on the screen, set up an array of Point objects representing your Polygon. Also create a Point object representing the origin of your Polygon.  

To test your code, you need to create instances of Asteroid objects in Asteroids. It is probably best to store any Asteroid objects you create as instance data, and initialize them in the constructor for Asteroids. Part I - Begin by having your Ship class implement the Java interface java.awt.event.KeyListener . When you add this to your Ship object using Eclipse, you should get a compile error where Eclipse asks you to add the unimplemented methods of the KeyListener interface. (The KeyListener API can be found at https://docs.oracle.com/javase/7/docs/api/java/awt/event/KeyListener.html )

You will have to implement logic in the keyPressed() and keyReleased() methods. (The keyTyped() method does not add functionality to our game, so it can simply have a return statement.)

When a key on your keyboard is pressed, the keyPressed() method is called automatically. Likewise, when a key is released, the keyReleased() method is invoked. You will use the up-arrow key for moving forward, the left-arrow for rotating left, and the right-arrow for rotating right. Logic for checking if the up-arrow key has been pressed or released appears as follows:

if(e.getKeyCode() == KeyEvent.VK_UP) {

// up arrow key

}

The names of the left and right arrow keys are KeyEvent.VK_LEFT and KeyEvent.VK_RIGHT respectively.

Perhaps the easiest way of solving this is to have three different boolean values (say, forward, left, and right that correspond to the three different movements. If the up arrow key is pressed, set the corresponding boolean value to true. If it is released, set it back to false. (Obviously all three boolean values would be initialized to false.) Therefore, if you are pressing on the up-arrow key, this boolean value is set to true. When you release the up-arrow key, the boolean is set to false.

To have the ship’s movements correspond to the arrow keys will involve modifying the move() method in the Ship class. As the move() method is called from paint(), it is called repeatedly. The move() method will now have to check the value of the three boolean values correspond to moving forward, or rotating left or right. If a ship is to move forward, you can use the following logic to set the value of the x and y coordinates:

position.x += 3 * Math.cos(Math.toRadians(rotation));

position.y += 3 * Math.sin(Math.toRadians(rotation));

(The rotation parameter is inherited from the Polygon class and is initialized when a Polygon object - such as a Ship - is created.) You will still need to also use your logic developed in Milestone #2 that has a ship reappear on the canvas if it moves off of the canvas.

For rotating left or right, you should notice that the Polygon class has a rotate() method that can be called for rotating left or right. If the boolean is set to rotating left or right, call the rotate() method and have it rotate a few degrees to the left or right. (The actual number of degrees may take some experimentation.)

Finally, you have to register the KeyListener in your Ship class with Asteroids. Add the following code in the constructor for the Asteroids class (the createShip() method is defined in the Asteroids class that can be downloaded for the Milestone #2 solution):

Part II - This involves detecting if an Asteroid has collided with your Ship. When you detect a collision, you will change the color of your Ship to some other color. After a short period  of time (i.e. one second) after a collision has occurred, set the color of the ship back to its original color.

This involves writing a collision() method in Polygon that checks if an Asteroid has collided with a Ship by detecting if a Ship is inside of an Asteroid object.

The collision() method can be written by simply getting the array of Point objects associated with a Polygon object, and then having a for loop go through this array of Point objects and see if they are contained in the other Polygon. (If the same Point object is contained in two different Polygons, there has been a collision.) For example, this method would appear as follows:

// Detect if there was a collision

public boolean collision(Polygon ship) {

        // get the Point objects for the Ship

Point[] points = ship.getPoints();

        for(Point p : points) {

if(this.contains(p)) {

return true;

}

}

return false;

}

You will notice the Polygon class has a contains() method that performs some “magical math” that determines if a Point is within (or inside of) another Polygon.

The Asteroids class then needs to check if there was a collision involving an Asteroid with the Ship. In the paint() method for Asteroids, there is a for loop that goes through each one of the random Asteroid objects that are created, invoking the paint() and move() methods for each Asteroid.  Add some code after calling move()  that calls the collision() method after moving an Asteroid (this will be called as asteroid.collision(ship) ). If a collision has occurred, you will have to set a boolean value that a collision has occurred to true. (Think of this as setting the collision status of the ship to true.)

After exiting this for loop, check the value of your boolean collision status. If a collision has occurred (i.e. your boolean is set to true), change the color of your Ship to another color (i.e. red). If a collision has not occurred, set the color to its standard color (i.e. magenta.)

One technique for maintaining the color of the ship (and the collision status) for a period of time is to set an integer value to some value (say 100). Every time  the paint() method is called in Asteroids, if a collision has occurred (the boolean is true), decrement this count. When the count goes to zero set the color of the Ship back to magenta and the value of the boolean that represents collisions back to false.

Test your code thoroughly that you can detect when an Asteroid has collided with your Ship, and that you set the appropriate color of the Ship.

. . .

Do not proceed to the following step until you are sure your ship can correctly fire bullet objects.

Java is a garbage-collected language which means that when objects are no longer able to be referenced, the garbage collector reclaims memory. However, unlike the Ship as well as Asteroid objects, when a bullet goes off of the screen, it does not reappear on the other side of the screen. A Bullet object that goes off of the screen is no longer visible, but it can still be referenced, so it will never be reclaimed by the garbage collector. These objects still occupy (and potentially waste) memory, but they can no longer be used. In computer systems we refer to this situation as a memory leak. A program that suffers from memory leaks can quickly waste large amounts of memory

To address memory leaks, The Bullet class will also have to determine if a bullet goes off the screen. Write a method called outOfBounds() in the Bullet class that returns true if a Bullet goes off of the screen. In the paint() method in Asteroids, after calling the paint() and move() methods of a Bullet, check if the bullet has gone off the screen by calling its outOfBounds() method. Identify the Bullet objects that need to be removed if they have gone off of the screen. After exiting the loop that paints and moves bullets, remove the Bullet objects that have gone off of the screen from the ArrayList of bullets. (By removing them from the ArrayList, they are no longer referenced and are now eligible for garbage collection.)

This last step will not be apparent from running your program, so you may need to use the debugger to ensure Bullet objects that have gone off of the screen are being removed from memory.

Detecting Bullets Hitting an Asteroid

The final step of completing the Asteroids game  will involve detecting when an asteroid has been hit by a bullet, as well as when the ship collides with an asteroid. If a ship shoots all of the asteroids, the player wins. If the ship has too many collisions with asteroids, the player loses.

In the paint() method of the Asteroids class you have a for loop that tracks bullets, checking to see if they have gone beyond the bounds of the screen. Add some code to this section that checks if a bullet and asteroid have collided. To do this, first add a getCenter() method to the Bullet class that returns the center Point of the Bullet object. Next, as you track each bullet object, you will have to determine if an Asteroid object has collided with a Bullet object. You will notice the Polygon class has a contains() method that is passed a Point object, and returns a boolean indicating if that Point is contained within the Polygon. Using this contains() method, along with your getCenter() method from the Bullet class, determine if a bullet and asteroid have collided. Keep track of each Asteroid that has collided with a bullet (you could put these in an ArrayList) and then upon exiting the for loop that checks for collisions, removes the Asteroid objects that collided with the Bullet objects. Additionally, the Bullet object that collided with the Asteroid must be removed as well.

Determining a Winner

At the end of the paint() method, check if there are any remaining asteroids. If not, change the screen to all black* and print out a message you won. At the top of the paint() method in Asteroids there is code that displays the value of counter. You can use similar code to print out the message that the player has won. *You may find the code at the very top of the paint() method that sets the color of the brush object, as well as a call to fillRect() on the brush object, helpful when setting the screen to all black.
